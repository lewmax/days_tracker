# DaysTracker Workflow

Цей документ описує, як працювати над DaysTracker від ідеї фічі до релізу, використовуючи Cursor‑агентів, Linear і Flutter‑інструменти.

## Організація slash-команд Cursor

Файли команд лежать у `.cursor/commands/` у вигляді `<номер кроку>-<slug>.md` і викликаються в чаті як `/<номер кроку>-<slug>`. **Номер збігається з розділом цього документа** (кроки 1–6, 9–10; кроки 7–8 і 11–12 без окремої команди).

Окремо: **`/0-review-config`** — аудит `.cursor/` (rules, skills, agents, commands); це не етап продуктового флоу.

## 1. Product & UX Brief

Мета: сформувати бачення фічі, користувачів і сценаріїв.

Кроки:
- Створити/оновити таску в Linear.
- Запустити в Cursor команду `/1-generate-design-brief`.
- Агенти й скіли:
  - **Product & UX Strategist** — веде ресерч і формує бриф.
  - **Skills**: `daystracker-domain`, `daystracker-design-principles`.
- Результат:
  - `docs/01_research.md` — конкурентний/UX ресерч.
  - `docs/02_design_brief.md` — продуктово‑UX бриф з основними флоу.

## 2. Дизайн‑система та екрани

Мета: зробити консистентний дизайн‑фреймворк і набір екранів.

Кроки:
- Після оновлення брифу запустити `/2-generate-design-from-brief`.
- Агенти й скіли:
  - **Design System & UX Designer**.
  - **Skill**: `daystracker-design-principles`.
- Що робить агент:
  - Проектує дизайн‑tokens (кольори, типографія, spacing, радіуси, стейти).
  - Визначає екрани, компоненти, IA та основні стани (empty/loading/error/normal).
  - Дає рекомендації по Penpot (сторінки, фрейми, компоненти).
- Результат:
  - `docs/design/03_design_tokens.md`.
  - `docs/design/04_screens_and_components.md`.
  - Необов’язкові нотатки для Penpot.

## 3. Tech Spec

Мета: перекласти бриф + дизайн у технічну архітектуру та feature‑спеки.

Кроки:
- Запустити `/3-generate-tech-spec`.
- Агенти й скіли:
  - **Tech Spec & Backlog Architect**.
  - **Skills**: `daystracker-domain`, дизайн‑скіли.
- Що робить агент:
  - Описує архітектуру (шари, навігація, state mgmt, error handling).
  - Визначає доменні сутності, інваріанти, enums.
  - Створює по одній feature‑спеці на фічу з user stories, сценаріями, acceptance criteria.
- Результат:
  - `docs/tech/architecture.md`.
  - `docs/tech/domain_model.md`.
  - `docs/features/*.md`.

## 4. Backlog & планування

Мета: перетворити спеки на структурований беклог у Linear.

Кроки:
- Запустити `/4-sync-backlog`.
- Агент: **Tech Spec & Backlog Architect**.
- Що робить агент:
  - Читає `docs/features/*.md`.
  - Групує роботу в епіки, сторі та сабтаски.
  - Пропонує статуси (Backlog v2/v1, Analysis, Ready for dev) і labels.
- Результат:
  - Оновлений беклог у Linear, пов’язаний з feature‑доками.

## 5. Bootstrap / інфраструктура (не завжди)

Мета: створити або оновити структуру Flutter‑проєкту під узгоджену архітектуру.

Використовувати коли:
- проєкт тільки запускається, або
- змінюється стек / основна архітектура.

Кроки:
- Запустити `/5-bootstrap-project`.
- Агент: **Bootstrap Architect**.
- Що робить агент:
  - Проводить Discovery (питання по стеку, DI, навігації, БД, тестам).
  - Пропонує й узгоджує архітектурний документ.
  - Створює/оновлює структуру `lib/`, DI, router, theme, базовий слайс.
- Результат:
  - Оновлений `pubspec.yaml`.
  - Базова структура `lib/` (core/domain/data/presentation).
  - Starter‑фіча як зразок.

## 6. Вибір таски

Мета: взяти конкретну фічу в роботу.

Кроки:
- В Linear перевести таску в `Ready for dev`.
- У Cursor запустити `/6-implement-current-task`.
- Агент: **Dev Feature Agent**.
- Що робить агент при старті:
  - Підтверджує ID таски та ціль.
  - Переводить її в `In Code` (згідно Process Rules).
  - Створює гілку `feat-{taskId}-{slug}`.

## 7. Аналіз & план фічі

Мета: чітко зрозуміти вимоги й домовитись про план реалізації.

Кроки:
- Dev Feature Agent читає:
  - `docs/features/<feature>.md`.
  - `docs/tech/architecture.md`, `docs/tech/domain_model.md`.
  - дизайн‑доки (`docs/design/*.md`).
- Агент формує:
  - список сценаріїв, які має покрити фіча,
  - відкриті питання,
  - короткий план (які шари й файли будуть змінені).
- Ти відповідаєш на питання й апрувиш або коригуєш план.

Поки критичні питання не закриті й план не погоджено — код не пишеться.

## 8. Реалізація фічі

Мета: імплементувати вертикальний slice, який відповідає спекам.

Кроки:
- Dev Feature Agent реалізує зміни в:
  - **domain** (сутності, репозиторії, сервіси),
  - **data** (імплементації, DTO, мапери),
  - **presentation** (BLoC/Cubit, віджети, навігація).
- За потреби оновлює `docs/features/*.md`.
- Запускає або просить запустити локальні Flutter‑команди з `docs/COMMANDS.md`:
  - `fvm dart analyze`, `fvm dart run custom_lint`, `fvm dart fix --apply`, скрипт `pre_commit` тощо.
- Готує чорновий опис PR: що зроблено, як це покриває acceptance criteria, які мануальні тести виконані.

Результат: готовий feature‑branch із реалізованою фічею.

## 9. Архітектурний чек

Мета: перевірити, що архітектурні правила не порушені.

Кроки:
- Запустити `/9-architecture-check` для поточного diff/PR.
- Агент: **Flutter Architecture Guard**.
- Що перевіряє:
  - відповідність Clean Architecture (domain/data/presentation),
  - DTO vs доменні моделі, мапінги,
  - BLoC‑boundaries (жодного сирого HTTP/SQL в UI),
  - роботу з `DateTime` і UTC,
  - чи треба оновити `architecture.md`/`domain_model.md`/`docs/features/*.md`.
- Результат:
  - звіт з blocking / strongly recommended / optional пунктами.

Blocking‑пункти треба виправити перед мерджем.

## 10. Code review & refactor

Мета: зробити код читабельним і підтримуваним без зміни поведінки.

Кроки:
- Запустити `/10-review-pr`.
- Агент: **Refactor/Review Agent**.
- Що робить:
  - дає high‑level враження по PR,
  - знаходить складні місця, поганий неймінг, дублікати, зайві залежності,
  - пропонує маленькі safe‑refactors (extract method/widget, rename, split клас).
- Ти приймаєш/відхиляєш пропозиції або просиш Dev Feature Agent застосувати їх у гілці.

## 11. Людський рев’ю та merge

Мета: фінальне підтвердження якості.

Кроки:
- Переглянути код, зміни в доках і звіти агентів.
- Потестити критичні сценарії руками на девайсі/емуляторі.
- Змерджити PR у `main`.
- Оновити статус таски в Linear на `Done`.

## 12. Release & операційка

Мета: зібрати та випустити реліз, оновити інфраструктурні речі.

Кроки:
- Використовувати команди з `docs/COMMANDS.md` для:
  - збірки (scripts `build_all`, `build_android`, `build_ios`),
  - bump версії (`version_bump`),
  - оновлення іконок, splash, локалізації.
- Оновити CHANGELOG і, за потреби, реліз‑ноти.

---

Цей флоу можна адаптувати під розмір фічі: для дрібних змін частину етапів (повний бриф, bootstrap) можна пропускати, але етапи аналізу, реалізації, архітектурного чеку й рев’ю бажано виконувати завжди.
